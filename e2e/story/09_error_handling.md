# 09. エラーハンドリングストーリー

## 概要
システムのエラーハンドリング機能をテストするE2Eテストストーリー

## テスト対象API
- error.yml

## ストーリー

### 1. テスト用エラーエンドポイント
- [ ] **GET /errors/internal-server-error** - 内部サーバーエラーを返却
  - ステータスコード500の確認
  - エラーレスポンス形式の確認
  - エラーメッセージの確認
  - エラータイプの確認

## テストシナリオ例

### シナリオ1: 内部サーバーエラーのテスト
1. テスト用エラーエンドポイントを呼び出し
2. HTTPステータスコード500が返却されることを確認
3. エラーレスポンスの構造を確認
   - `type` フィールドの存在確認
   - `message` フィールドの存在確認
   - `data` フィールドの存在確認（オプション）
4. エラーメッセージの内容確認
5. エラータイプが適切に設定されていることを確認

### シナリオ2: エラーレスポンス形式の統一性確認
1. 各APIで発生する可能性のあるエラーパターンを確認
2. 400エラー（Bad Request）のレスポンス形式確認
3. 401エラー（Unauthorized）のレスポンス形式確認
4. 404エラー（Not Found）のレスポンス形式確認
5. 500エラー（Internal Server Error）のレスポンス形式確認
6. 全てのエラーレスポンスが統一された形式であることを確認

### シナリオ3: エラーハンドリングの網羅的テスト
1. 無効なパラメータでの各API呼び出し
2. 存在しないリソースIDでの各API呼び出し
3. 権限不足での各API呼び出し
4. 必須パラメータ不足での各API呼び出し
5. 各エラーケースで適切なステータスコードが返却されることを確認

## 前提条件
- 基本セットアップが完了している
- 各種APIが正常に動作している
- 認証設定が適切に行われている

## 期待結果
- テスト用エラーエンドポイントが正常に500エラーを返却する
- エラーレスポンスの形式が統一されている
- 各エラーケースで適切なステータスコードが返却される
- エラーメッセージが適切に設定されている
- エラータイプが正しく分類されている

## 注意事項
- このエンドポイントはテスト用途のみで使用する
- 本番環境では無効化される可能性がある
- エラーハンドリングのテストは他のAPIの動作に影響しないよう注意する

## エラーレスポンス形式
```json
{
  "type": "error_type",
  "message": "Error message",
  "data": {
    "sign_in_page_url": "auth.dev.saasus.io"
  }
}
```

## 共通エラーパターン
- **400 Bad Request**: 無効なパラメータ、必須パラメータ不足
- **401 Unauthorized**: 認証失敗、無効なトークン
- **404 Not Found**: 存在しないリソース
- **500 Internal Server Error**: サーバー内部エラー

## テスト対象外
- ネットワークエラー（タイムアウト、接続エラー等）
- クライアント側のエラーハンドリング
- レート制限エラー（実装されている場合）
