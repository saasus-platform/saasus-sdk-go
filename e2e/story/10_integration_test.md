# 10. 統合テストストーリー

## 概要
全てのAPI機能を組み合わせた統合的なE2Eテストストーリー

## テスト対象API
- 全てのAPI（authInfo.yml, basicInfo.yml, credential.yml, env.yml, error.yml, invitation.yml, role.yml, saasUser.yml, singleTenant.yml, tenant.yml, tenantAttribute.yml, tenantUser.yml, userAttribute.yml, userInfo.yml）

## ストーリー

### 1. 完全なSaaSセットアップフロー
- [ ] **基本設定の完了**
  - ドメイン設定、認証設定、環境設定
  - カスタマイズページ設定
  - 通知メールテンプレート設定
- [ ] **属性定義の作成**
  - ユーザー属性定義の作成
  - テナント属性定義の作成
- [ ] **役割(ロール)の定義**
  - 管理者、編集者、一般ユーザー、閲覧者役割の作成
- [ ] **テナントの作成と設定**
  - 複数テナントの作成
  - テナント属性の設定
  - 請求情報の設定

### 2. ユーザー管理の統合フロー
- [ ] **SaaSユーザーの作成**
  - 新規登録フローでのユーザー作成
  - 管理者による直接ユーザー作成
  - AWS Marketplace連携ユーザー作成
- [ ] **テナントユーザーの管理**
  - ユーザーのテナント追加
  - 環境毎の役割付与
  - ユーザー属性の設定
- [ ] **招待機能の活用**
  - 新規ユーザーの招待
  - 既存ユーザーの別テナント招待
  - 招待の検証と承認

### 3. 認証・認可の統合テスト
- [ ] **認証フローの確認**
  - ログイン処理
  - 認証情報の保存・取得
  - リフレッシュトークンによる更新
- [ ] **ユーザー情報の取得**
  - IDトークンからのユーザー情報取得
  - メールアドレスからのユーザー検索
  - テナント・役割情報の確認
- [ ] **権限確認**
  - 環境毎の役割に基づく権限確認
  - 複数テナント所属時の権限分離

### 4. 高度な機能の統合テスト
- [ ] **外部IDプロバイダ連携**
  - Google認証の設定と利用
  - SAML認証の設定と利用（テナント毎）
- [ ] **MFA機能**
  - MFA設定の有効化
  - 認証アプリの登録
  - MFA認証フロー
- [ ] **シングルテナント機能**
  - シングルテナント設定
  - CloudFormation連携
  - AWS環境との統合

### 5. エンドツーエンドシナリオ

#### シナリオA: 新規SaaSの立ち上げ
1. 基本設定の完了（ドメイン、認証、メール等）
2. 役割定義の作成（admin, user, viewer）
3. 環境設定（dev, staging, prod）
4. 最初のテナント作成
5. 管理者ユーザーの作成と役割付与
6. 招待機能での追加ユーザー招待
7. 各ユーザーでのログインと権限確認

#### シナリオB: 企業向けマルチテナントSaaS
1. 複数テナントの作成（企業A、企業B、企業C）
2. テナント毎の外部IDプロバイダ設定（SAML）
3. 各テナントでのユーザー管理
4. 環境毎の役割設定（本番、開発環境での権限分離）
5. テナント間でのユーザー移動
6. 請求情報の管理

#### シナリオC: AWS Marketplace連携SaaS
1. AWS Marketplace連携の設定
2. Marketplace経由での新規ユーザー登録
3. 既存テナントとの連携
4. Stripe連携での課金管理
5. プラン変更フロー

#### シナリオD: セキュリティ重視のSaaS
1. 強固なパスワードポリシー設定
2. MFA必須設定
3. デバイス記憶設定
4. 外部IDプロバイダとの連携
5. セキュリティ監査ログの確認

### 6. パフォーマンステスト
- [ ] **大量データでのテスト**
  - 100ユーザー、10テナントでの動作確認
  - 大量招待の処理
  - 複雑な権限設定での応答時間確認
- [ ] **同時アクセステスト**
  - 複数ユーザーの同時ログイン
  - 同時招待処理
  - 同時設定変更

### 7. 障害復旧テスト
- [ ] **エラー状況での動作確認**
  - 一時的なサービス停止からの復旧
  - 不正なデータでの処理
  - ネットワーク障害時の挙動

## 前提条件
- 全ての個別機能テストが完了している
- テスト環境が適切に設定されている
- 必要な外部サービス（AWS、Stripe等）との連携が可能

## 期待結果
- 全ての機能が統合的に動作する
- 実際のSaaS運用シナリオが正常に実行できる
- パフォーマンスが許容範囲内である
- エラー時の適切な処理が行われる
- セキュリティ要件が満たされる

## 成功基準
- [ ] 全てのAPIが正常に動作する
- [ ] 統合シナリオが100%成功する
- [ ] パフォーマンス要件を満たす
- [ ] セキュリティテストに合格する
- [ ] エラーハンドリングが適切に動作する

## 注意事項
- 統合テストは時間がかかるため、適切なタイムアウト設定が必要
- テストデータの準備と後処理を適切に行う
- 外部サービス連携時は課金に注意する
- テスト環境と本番環境の分離を徹底する
